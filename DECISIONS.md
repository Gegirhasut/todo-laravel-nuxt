# Инженерные решения

Здесь собраны решения, которые в коде видны только косвенно: как трактованы неоднозначности ТЗ, почему выбраны эти инструменты, какие компромиссы приняты сознательно и что так же сознательно оставлено за рамками. README остаётся практическим — как запустить и что где лежит.

---

## 1. Видимость задач и осознанный IDOR-трейд-офф

> [!IMPORTANT]
> ТЗ внутренне противоречиво в вопросе видимости задач. Раздел про роли (необязательный) говорит «пользователь видит только свои задачи», при этом таблица эндпоинтов помечает `GET /api/tasks/{id}` как просто «auth» — и ограничивает владельцем/админом только `PUT/PATCH/DELETE`. Оба прочтения одновременно верными быть не могут, поэтому выбор зафиксирован явно.

Принятая трактовка:

| Действие | Кто может | Почему так |
| --- | --- | --- |
| Список `GET /tasks` | пользователь — только свои, админ — все | это и есть «ролевое» требование ТЗ |
| Одна задача `GET /tasks/{id}` | любой авторизованный | в таблице эндпоинтов доступ помечен «auth»; заодно работают прямые ссылки на задачу |
| `PUT/PATCH/DELETE /tasks/{id}` | владелец или админ | как в таблице эндпоинтов; сервер отвечает `403` |

Права навязывает сервер (`TaskPolicy`, `403`), а UI их только отражает: у чужой задачи, открытой по прямой ссылке, кнопок «Изменить» и «Удалить» просто нет. Это одновременно закрывает пункт ТЗ «скрывать кнопки редактирования и удаления, если нет доступа» — он осмыслен ровно в сценарии, когда не-владелец открывает чужую задачу.

Честная оговорка: открытое чтение по id — это классический broken object level authorization (IDOR). При последовательных числовых id залогиненный пользователь может перебором прочитать чужие задачи, которые список от него скрывает. Выбрал это сознательно, а не по недосмотру — следуя букве таблицы ТЗ (`{id}` = auth) и ради работоспособности расшариваемых ссылок на задачу.

В реальном проде закрыл бы одним из двух способов: (1) ограничить `view()` до owner-or-admin, согласовав чтение с `update`/`delete`; либо (2) если открытое чтение действительно нужно продукту — перейти на неугадываемые идентификаторы (ULID) вместо автоинкремента и убирать `owner` из ответа для не-владельцев. В обоих случаях правило видимости стоит вынести в единый scope `Task::visibleTo(User)`, чтобы список и чтение задач гарантированно не расходились.

## 2. Аутентификация: Sanctum Bearer-токены, а не cookie-сессия

SPA живёт на другом origin (`:3000`), чем API (`:8000`). Cookie-режим Sanctum в такой схеме требует согласованных `stateful_domains` / `SESSION_DOMAIN`, отдельного запроса за CSRF-токеном и аккуратных same-site настроек — всё это хрупко при разных origin и внутри Docker. Stateless bearer-токен от origin не зависит и естественно ложится на decoupled SPA.

Практические следствия: токен создаётся на `POST /auth/login`, хранится в cookie только ради переживания перезагрузки страницы, каждый запрос уходит с `Authorization: Bearer`, любой `401` чистит сессию и уводит на `/login`. Logout отзывает только текущий токен — входы на других устройствах живут дальше.

## 3. Laravel 12, а не 13

ТЗ разрешает PHP 8.2+. Laravel 13 требует PHP 8.3, Laravel 12 — текущая стабильная ветка с поддержкой 8.2. Выбор 12 — это не «не успел обновиться», а следование заявленной в ТЗ нижней границе платформы.

---

## 4. SPA, а не SSR

Всё содержимое приложения — персональные данные за авторизацией: их нельзя закешировать на CDN и некому отдавать анонимно. Публичного контента, ради которого существуют SSR/SSG (SEO, Open Graph, быстрый первый рендер для гостей), здесь нет — единственная открытая страница `/login` не индексируется. SSR добавил бы Node-сервер рядом с PHP и удвоил путь до API (браузер → Node → Laravel), не дав ничего взамен.

> [!NOTE]
> SSR технически включается одним флагом (`ssr: true`), но одного флага мало — придётся пересмотреть слой авторизации: явно прокидывать токен в серверные вызовы API (`useRequestHeaders`), перевести загрузку данных на `useAsyncData`, сделать cookie `httpOnly` (что тянет за собой CSRF-требования cookie-режима Sanctum) и следить, чтобы токен не сериализовался в SSR-пейлоад. Решение «включить SSR» — это на самом деле решение «переделать auth», и для этого приложения оно не окупается.

## 5. Сортировка по статусу — жизненный цикл, а не алфавит

Алфавитная сортировка enum-строк ставит `completed` первым — для задачника это бессмысленно. Сортировка идёт по жизненному циклу: `in_progress → pending → completed` (активное сверху, завершённое внизу; `desc` — наоборот).

Приоритет задан в одном месте — `TaskStatus::sortOrder()` в enum, а SQL-выражение `CASE status WHEN … THEN …` строится из него. Изменение порядка — правка одного `match` в enum, а не поиск магических чисел по запросам. В базе и в API статус остаётся строкой: контракт ТЗ (enum-значения) не тронут, порядок сортировки — деталь представления, а не хранения.

## 6. Масштабируемость: где узкое место на самом деле

> [!TIP]
> Сортировка через `CASE` — не проблема, и оптимизировать её заранее не нужно: выражение вычисляется на уже отфильтрованном по `user_id` и пагинированном наборе, а составной индекс `(user_id, status)` кроет и фильтр, и выборку.

Первым реальным узким местом при росте данных станет поиск `LIKE '%…%'` — он не использует индексы по определению (ведущий `%`). Порядок действий, когда это станет заметно: FULLTEXT-индекс в MySQL / `tsvector` + GIN в PostgreSQL, а при требованиях к релевантности и опечаткам — внешний движок (Meilisearch/Typesense через Laravel Scout). До этого момента любой из этих шагов был бы преждевременной сложностью.

## 7. Безопасность

- **Единый JSON-контракт ошибок** (`401/403/404/422/500`) в одном обработчике — клиенту не приходится парсить HTML-страницы ошибок, а формат не расползается по контроллерам.
- **Политики поверх UI-гейтинга**: скрытые кнопки — вежливость, `TaskPolicy` — граница. Прямой запрос к чужой задаче получает `403` независимо от того, что показывал интерфейс.
- **Rate limit на логине и на API**: логин — 6 попыток в минуту на пару email+IP, дальше `429`; авторизованная группа эндпоинтов — 60 запросов в минуту на пользователя, чтобы валидный токен не мог без ограничений молотить поиск с ведущим `%`.
- **Токены с истечением** (7 дней) и ежедневный прунинг просроченных — утёкший токен ограничен во времени, таблица токенов не растёт бесконечно.
- **Никакой утечки чужих данных**: `TaskResource` отдаёт владельца без email; владелец задачи всегда берётся из авторизованного пользователя, а не из тела запроса.

## 8. Сознательно за рамками ТЗ

Скоуп тестового удержан намеренно: всё ниже — то, что имело бы смысл в реальном продукте, но здесь добавило бы поверхности для ревью, не показав ничего нового о качестве работы.

- **Rich-text / Markdown в описании.** Сейчас описание — plain text (`white-space: pre-line`), XSS-поверхности нет вовсе. В продукте: хранить markdown-исходник как есть, рендерить через санитизацию с двух сторон — DOMPurify на клиенте и серверная очистка HTML при отдаче.
- **SSR/SSG** — см. раздел 4: осмысленно только вместе с пересмотром авторизации.
- **Полнотекстовый поиск** — см. раздел 6: первый шаг при реальном росте данных.
- **Переключатель «Мои / Все» для админа** — фильтр списка по владельцу; сейчас админ видит все задачи с пометкой «Моя» на своих.
- **Продакшен-профиль Docker.** Текущая сборка сознательно dev-grade и рассчитана на локальное ревью: `php artisan serve` и dev-сервер Nuxt, сид базы при каждом старте, `APP_DEBUG=true`, `chmod` на storage. Продакшен-профиль — php-fpm/nginx или FrankenPHP, собранный вывод Nuxt, `APP_ENV=production` без debug, точный список CORS-origin вместо шаблона приватных сетей, TLS — осознанно за рамками тестового.

> [!NOTE]
> **Сверх ТЗ уже сделано** (кратко, без подробностей): постоянные ссылки на задачи (`/tasks/{id}` работает и поверх списка, и как отдельная страница), светлая/тёмная темы с фирменным акцентом, редактирование карточки на месте в два чётких режима (чтение/форма), кастомные выпадающие элементы под тему вместо системных, осмысленная сортировка по статусу, доступность (фокус-трапы, aria-атрибуты, скрытие ссылок для скринридеров). Основной сигнал этого документа — не список фич, а то, как принимались решения.
